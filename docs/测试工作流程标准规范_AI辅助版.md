## 1. 目的与范围

### 1.1 目的

本文档旨在规范测试岗位工作流程，明确各阶段工作标准，并引入 AI 辅助测试的方法与规范，提升测试效率与质量，实现测试工作的标准化、智能化。

### 1.2 适用范围

- 功能测试
- 接口测试
- 自动化测试
- 回归测试

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| AI 辅助测试 | 利用 AI 工具辅助完成测试用例生成、测试数据准备、测试报告生成等工作的测试模式 |
| 测试用例 | 为某个测试目标而编写的 一组测试输入、执行条件和预期结果的文档 |
| 缺陷 | 软件中存在的错误、遗漏或多余功能 |
| 覆盖率 | 测试用例对需求的覆盖程度 |

## 3. 各阶段工作规范

### 3.1 需求分析

#### 3.1.1 标准动作

1. 接收需求文档（PRD/原型/接口文档）
2. 理解业务逻辑和功能要求
3. 提取测试点和测试范围
4. 识别需求中的模糊点和遗漏风险
5. 输出需求分析报告

#### 3.1.2 AI 辅助规范

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| 需求分析 | 需求文档/原型截图 | 测试点列表、边界条件、潜在风险 | 覆盖核心功能点 |

**AI 辅助示例**：

```
输入：内测邀请页面 URL 或需求描述
输出：
  - 功能测试点：邀请码输入、验证码、提交按钮
  - 边界条件：空输入、超长输入、特殊字符
  - 异常场景：网络超时、重复提交
```

#### 3.1.3 交付物

- 需求分析报告
- 测试点清单

---

### 3.2 测试计划

#### 3.2.1 标准动作

1. 确定测试范围和目标
2. 评估测试工作量
3. 分配测试资源和时间
4. 制定风险应对策略
5. 输出测试计划文档

#### 3.2.2 AI 辅助规范

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| 工作量估算 | 需求规模、用例数量 | 预计工时、人员分配 | 偏差 ≤ 20% |
| 风险识别 | 项目信息 | 潜在风险列表 | 覆盖常见风险 |

#### 3.2.3 交付物

- 测试计划文档（版本、范围、资源、进度、风险）

---

### 3.3 用例设计

#### 3.3.1 标准动作

1. 根据测试点设计测试用例
2. 设定用例优先级（P0/P1/P2）
3. 明确前置条件、测试步骤、预期结果
4. 用例格式规范化
5. 纳入用例库管理

#### 3.3.2 AI 辅助规范（核心环节）

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| **用例生成** | 需求/页面/接口文档 | 结构化测试用例（MD/XMind/Excel） | 覆盖率 ≥ 80% |
| **用例补充** | 已有用例 | 补充边界用例、异常用例 | 遗漏率 ≤ 10% |
| **用例优化** | 旧有用例 | 优化建议、去重 | 提升可维护性 |

**用例模板规范**：

| 字段 | 说明 | 必填 |
|------|------|------|
| 用例ID | 唯一标识（如 TC-001） | ✅ |
| 用例名称 | 清晰描述测试目标 | ✅ |
| 前置条件 | 执行前需满足的条件 | ✅ |
| 测试步骤 | 详细的操作步骤 | ✅ |
| 预期结果 | 期望的正确结果 | ✅ |
| 优先级 | P0/P1/P2 | ✅ |
| 备注 | 注意事项、关联需求 | - |

#### 3.3.3 AI 生成用例示例

**输入**：邀请码页面 URL + 需求描述

**输出**：

```
## TC-001: 页面正常加载
| 项目 | 内容 |
|------|------|
| 用例ID | TC-001 |
| 用例名称 | 页面正常加载 |
| 前置条件 | 网络连接正常 |
| 测试步骤 | 打开邀请链接，等待页面加载完成 |
| 预期结果 | 页面正常显示，内容完整无缺失 |
| 优先级 | P0 |

## TC-002: 有效邀请码验证
...
```

#### 3.3.4 交付物

- 测试用例库（Markdown/XMind/Excel 格式）
- 用例统计表（总数、优先级分布）

---

### 3.4 用例评审

#### 3.4.1 标准动作

1. 组织评审会议（测试组 + 产品 + 开发）
2. 检查用例完整性、正确性、可执行性
3. 识别遗漏和冗余
4. 记录评审意见
5. 更新用例

#### 3.4.2 AI 辅助规范

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| 用例检查 | 测试用例 | 遗漏场景、逻辑错误、冗余建议 | 辅助发现 ≥ 30% 问题 |

#### 3.4.3 交付物

- 评审记录
- 更新后的测试用例

---

### 3.5 测试执行

#### 3.5.1 标准动作

1. 准备测试环境
2. 按照用例执行测试
3. 记录执行结果（通过/失败/阻塞）
4. 提交缺陷
5. 更新用例状态

#### 3.5.2 执行顺序

```
P0 用例（核心功能）→ P1 用例（重要功能）→ P2 用例（边界/界面）
```

#### 3.5.3 AI 辅助规范

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| 测试数据生成 | 用例需求 | 测试数据（账号、参数、边界值） | 符合用例要求 |
| 自动化脚本 | 测试用例 | 自动化脚本代码 | 可执行率 ≥ 70% |
| 回归分析 | 缺陷列表 | 推荐回归用例 | 覆盖率 ≥ 90% |

#### 3.5.4 交付物

- 测试执行记录
- 缺陷报告

---

### 3.6 缺陷管理

#### 3.6.1 标准动作

1. 提交缺陷（标题、描述、重现步骤、影响范围）
2. 跟踪缺陷修复状态
3. 验证缺陷修复
4. 关闭缺陷
5. 统计缺陷数据

#### 3.6.2 缺陷分级标准

| 级别 | 定义 | 例子 |
|------|------|------|
| P0 | 阻断：功能完全不可用 | 闪退、无法登录、数据丢失 |
| P1 | 严重：核心功能异常 | 关键流程走不通、数据显示错误 |
| P2 | 一般：非核心功能问题 | 界面样式、提示文案错误 |
| P3 | 轻微：不影响使用 | 排版、字体、颜色等 |

#### 3.6.3 AI 辅助规范

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| 缺陷分析 | 缺陷描述 | 缺陷分类、根因分析 | 辅助定位 |
| 修复建议 | 缺陷信息 | 可能原因、修复方案参考 | 仅供参 |

#### 3.6.4 交付物

- 缺陷报告
- 缺陷统计表

---

### 3.7 测试报告

#### 3.7.1 标准动作

1. 汇总测试执行数据
2. 统计缺陷情况
3. 计算测试覆盖率
4. 评估质量风险
5. 输出测试报告

#### 3.7.2 AI 辅助规范（核心环节）

| AI 能力 | 输入 | 输出 | 质量要求 |
|---------|------|------|----------|
| **报告生成** | 执行数据、缺陷数据 | 结构化测试报告 | 完整率 100% |
| **数据分析** | 测试数据 | 趋势分析、风险提示 | 准确率 ≥ 90% |
| **摘要提取** | 完整报告 | 执行摘要（一页版） | 关键信息不遗漏 |

#### 3.7.3 测试报告模板


#### 3.7.4 交付物

- 测试报告（详细版）

## 7. 流程检查点

### 7.1 关键检查点

| 阶段 | 检查点 | 责任人 | 通过标准 |
|------|--------|--------|----------|
| 用例设计 | AI 用例生成完成 | 测试工程师 | 覆盖率 ≥ 80% |
| 用例评审 | 评审通过 | 测试组 | 问题已解决 |
| 测试执行 | 执行完成 | 测试工程师 | 执行率 100% |
| 测试报告 | 报告审批 | 测试组长 | 数据准确完整 |

### 7.2 质量门禁

- 用例评审通过率 < 80% → 禁止进入测试执行
- P0/P1 缺陷未清零 → 禁止上线
- 测试报告未通过审批 → 禁止归档
